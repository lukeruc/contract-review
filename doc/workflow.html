合同审核系统

基于 Claude Code 的 EPC 三层架构 · 5000 字符自动区分简单/复杂模式 · 产出审核意见书 + 修订合同 + 审核工作报告

核心架构

🏗

Architect（总承包）

唯一用户入口。不做第一手法律分析。负责判断、规划、委托、信息蒸馏、验收。

👷

Task Agent（分包商）

按需创建的 sub-agent，执行单项任务后销毁。3 个固定定义（Structure / Preliminary Report / Assembly），其余 Architect 自行撰写 task spec。

🛠

Tools + Reviewer

工具层为 Task Agent 提供信息获取和文件处理能力。Reviewer（仅复杂模式）逐条对照交付标准核查产出。合并忠实性抽查也由 Reviewer 执行。

⚖ 5000 字符阈值 — 纯机械判断

≤5000 → 简单模式（无 Reviewer）   |   >5000 → 复杂模式（EPC 全机制）

简单模式（≤5000 字符）

1

合同类型判断与规则匹配

Architect 读取 contract.md → 判断合同类型 → 扫描 rules/ 匹配规则文件。无匹配则上报用户选择。

2

审核

以规则为底本，根据合同内容和用户关切制定 delivery_standards。创建 Audit Agent（T-001），注入 yd-law + qcc，审查全合同。Architect 自行验收。

3

修订

Revision Agent（T-002）注入 docx-cli 全命令，逐条执行修改建议，每处修订附批注。Architect 逐条核验。

4

格式输出

审核意见书直接输出 .md。修订合同由 Format Agent 生成 .docx（含批注）。

5

交付

交付 audit-opinion.md + audit-work-report.md/.html + revised.docx。

6

对话续接 + 审核工作报告

Architect 编制 agent/CLAUDE.md + agent/index.md，并撰写 output/audit-work-report.md/.html。用户可在 agent/ 下新开对话追问。

Reviewer 审查体系（仅复杂模式）

Phase 1 — 初设阶段

  • 格式完整性、章节覆盖
  • 条款编号一致性
  • 数据引用可追溯
  • T-S01 特别：按索引表逐条搜索原文验证

形式审核，不做法律实质判断

Phase 2 — 详细审查 + 合并忠实性

  • 核验事实引用、条款关系、法条引用
  • 结论与合同结构一致性
  • 分层合并 Review：抽样 3×N 条，验证存在性、风险等级、描述完整性

单点实质性审查 + 合并忠实性核查，3 轮上限

关键设计决策

#决策
1仅通过 /contract-review 命令触发
2不接受 .doc 格式
35000 字符阈值纯机械判断（char-count.sh）
4审核立场在 bootstrap 阶段向用户确认
5审核规则是 delivery_standards 底本，Architect 根据合同内容和用户关切调整
6所有 Agent 间通讯使用 Markdown + YAML frontmatter
7Architect 不做第一手法律分析，全部委托 Task Agent
8工具以 skill 形式存在，按需注入 sub-agent
9Reviewer 仅复杂模式启用，循环上限 3 次
10简单模式不启用 Reviewer
11Structure / Preliminary Report / Assembly 使用固定定义，Architect 直接注入
12Assembly 分层合并：≤10万字单次，>10万字分组并行+Reviewer抽样
13审核意见书交付 .md，修订合同 .docx 每处修订附批注
14Reference 文件是参考示例，非强制模板
