合同审核系统

基于 Claude Code 的 EPC 三层架构 · 自动区分简单/复杂模式 · 产出审核意见书 + 修订合同

核心架构

🏗

Architect（总承包）

唯一用户入口。不做第一手法律分析。负责判断、规划、委托、信息蒸馏、验收。

👷

Task Agent（分包商）

按需创建的 sub-agent，执行单项任务后销毁。所有法律分析全部委托 Task Agent 执行。

🛠

Tools + Reviewer

工具层为 Task Agent 提供信息获取和文件处理能力。Reviewer（仅复杂模式）逐条对照交付标准核查产出。

⚖ 5000 字符阈值 — 纯机械判断

≤5000 → 简单模式（无 Reviewer）   |   >5000 → 复杂模式（EPC 全机制）

简单模式（≤5000 字符）

1

合同类型判断与规则匹配

Architect 读取 contract.md → 判断合同类型 → 扫描 rules/ 目录匹配规则文件。无匹配则上报用户选择。

2

审核 — 制定交付标准 + Audit Agent

以规则文件为底本，根据合同实际内容和用户关切制定 delivery_standards。创建单个 Audit Task Agent（T-001），注入 yd-law + qcc，审查全合同。

2

Architect 自行验收

对照合同原文和 delivery_standards 逐项验收审核意见书。不通过则重建 T-001。

3

修订 — Revision Agent

创建 Revision Task Agent（T-002），注入 docx-cli 全命令。逐条执行审核意见书的修改建议。Architect 逐条核验修订内容。

4

格式输出

Format Task Agents 将审核意见书和修订合同 Markdown 转为 .docx。

5

交付 + 对话续接

交付两份 .docx。Architect 编制 agent/CLAUDE.md（上下文快照）和 agent/index.md（文件索引），用户此后可在 agent/ 下新开对话追问。

qcc企业工商信息查询。4 大类 67 个工具：企业信息、经营动态、知识产权、风险扫描Task Agent（Audit）
char-count.sh纯文本字符数统计，用于 5000 字符阈值判断Architect（bootstrap）

Reviewer 双阶段审查（仅复杂模式）

Phase 1 — 初设阶段

上下文：system prompt + 工作要求 + 交付物

  • 格式是否正确
  • 章节是否完整覆盖
  • 条款编号是否与原文一致
  • 数据引用是否可追溯
  • T-S01 特别：按索引表逐条搜索原文验证

仅形式审核，不做法律实质判断

Phase 2 — 详细审查阶段

上下文：+ shared-context + Reviewer Briefing

  • 核验事实引用是否正确
  • 核验条款引用关系是否与联系报告吻合
  • 核验结论是否与合同结构明显矛盾
  • 核验法条引用是否明显错误

单点实质性审查，3 轮上限

对话续接

审查完成后，Architect 在会话目录下创建 agent/ 目录。用户在新 Claude Code 实例中打开此目录，CLAUDE.md 自动加载，Agent 即刻进入对话状态。

agent/CLAUDE.md

  • 合同速览（类型、当事方、交易结构）
  • 审查结论摘要（风险评级、核心发现）
  • 修订概况
  • 文件索引指引（所有产出位置）
  • 常见追问导航（预判问题+应读文件）
  • Agent 自我认知（定位+边界）

agent/index.md

  • 按目录分组列出所有文件
  • 每行：文件名、描述、产生者
  • Agent 据此了解项目全貌

关键设计决策

#决策
1仅通过 /contract-review 命令触发，不做自动语义匹配
2不接受 .doc 格式，要求用户转换为 .docx 或 PDF
35000 字符阈值纯机械判断，用 char-count.sh
4审核立场必须在 bootstrap 阶段向用户确认（代表方、审核目标、风险偏好）
5审核规则是 delivery_standards 的底本，不是成品。Architect 根据合同内容和用户关切调整
6所有 Agent 间通讯使用 Markdown + YAML frontmatter
7Architect 不做第一手法律分析，全部委托 Task Agent
8工具以 skill 形式存在，按需注入 sub-agent
9Reviewer 仅复杂模式启用，循环上限 3 次
10简单模式不启用 Reviewer，Architect 自行验收
11版本号在 frontmatter 中管理，不体现于文件名
12所有中间产物可追溯
13Structure Agent 使用固定定义，Architect 直接注入，输出含结构索引表；Reviewer 按索引表逐条搜索原文核查
14Reference 文件是参考示例，非强制模板
