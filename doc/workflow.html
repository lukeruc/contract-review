02 核心架构：三层协作模型

系统采用 总承包–分包–工具 三层架构。Architect 为用户唯一入口，不下场做第一手分析；Task Agent 按需创建、执行单项任务后销毁；复杂模式下 Reviewer 独立核查交付质量。

🏗

Architect（总负责人）—— 用户唯一入口

不做第一手法律分析。负责判断、规划、委托 Task Agent、信息蒸馏、最终验收。

👷

Task Agent（执行者）—— 按需创建，任务完成即销毁

核心任务类型：结构化、条件提取、交叉引用分析、条款审核、文本修订、格式输出。

🔍

Reviewer（质量监理）—— 仅复杂模式启用

独立 AI 实例，对照交付标准逐条核查 Task Agent 产出。不通过退回修订（上限 3 轮），通过放行。保护 Architect 上下文不被多个 Agent 交付物淹没。

🛠

工具层 —— Task Agent 的基础设施

文档转换（mdconverter）、文档编辑（docx-cli）、法律检索（yd-law）、企业查询（qcc）、文本统计（char-count.sh）。

1

Architect 读取合同，判断类型，扫描 rules/ 匹配规则文件。规则为底本，根据合同内容和用户关切调整 delivery_standards。无匹配则上报用户。

2

审核

创建 Audit Agent（T-001），注入 yd-law + qcc，审查全合同。Architect 自行验收（简单模式无 Reviewer）。

3

修订

Revision Agent（T-002）注入 docx-cli，逐条执行修改建议。修订人姓名由 shared-context.md 指定。Architect 逐条核验。

4

格式输出

审核意见书输出 .md。Format Agent（T-FMT-02）生成修订合同 .docx（含批注）。

5

交付

6

对话续接与工作报告

T-S02 Conditions —— 商业安排结构提取

与 contract-review 不同：不做"可核验数据清单"（模板中无硬数据），而是提取商业安排结构——付款如何分段、条件如何挂钩、风险如何配置。供 Question Generation Agent 建立对模板商业逻辑的全局理解。

T-S03 Cross-References —— 保护网分析

与 contract-review 不同：不做"冲突检测"（模板起草人已确保一致），而是分析起草方的保护网设计——哪些条款互相关联形成对一方的系统性保护，引用链条的核心节点在哪。

Question Generation Agent —— 问题生成引擎

设计要素设计内容
正向标准8 大类 36 条判断标准，覆盖经济利益、风险分配、单方权利、偏离法定、前置条件、期限终止、争议解决、合同结构
反向标准6 大类 20 条过滤规则（纯格式、标准程序、已平衡条款、背景陈述、技术标识），抑制无意义提问
数量上限硬上限 30 条，迫使 Agent 按重要性排序，仅最重要的条款进入清单
优先级涉及金钱 > 不涉及；单方权利 > 双方对等；偏离法定 > 与法定一致；不可事后修正 > 可事后补充；高频争议领域 > 低频
上下文策略直接注入 conditions + crossref + 结构索引表；结构化全文在磁盘上按需读取（避免上下文溢出）
审校立场通过上下文文件传递，不修改 System Prompt，确保每个问题带有明确的立场指向

07 Reviewer 审查体系（仅 contract-review 复杂模式）

Phase 1 —— 初设审查

  • 格式完整性检查
  • 章节覆盖验证
  • 条款编号一致性
  • 数据引用可追溯性
  • T-S01 特别机制：按结构索引表逐条搜索原文验证——每次检查为精确匹配

定位：形式审核，不做法律实质判断

Phase 2 —— 详细审查 + 合并忠实性

  • 事实引用是否正确
  • 条款关系是否与 crossref 报告吻合
  • 结论是否与合同结构明显矛盾
  • 分层合并 Review：从每组输入取 3×N 条抽检，验证风险等级、核心描述、无遗漏

定位：单点实质性审查 + 合并忠实性核查，循环上限 3 次

#决策适用范围

2不接受 .doc 格式，要求用户另存为 .docx两个 skill
310000 字符为建议阈值，用户最终决定简单/复杂模式contract-review
4审核立场在 bootstrap 阶段必须向用户确认，不设默认值两个 skill
5审核规则是 delivery_standards 底本，Architect 根据合同内容和用户关切调整contract-review
6所有 Agent 间通讯使用 Markdown + YAML frontmatter两个 skill
7Architect 不做第一手法律/模板分析，全部委托 Task Agent两个 skill
8工具以 skill 形式存在，按需注入 sub-agent两个 skill
9Reviewer 仅 contract-review 复杂模式启用，循环上限 3 次contract-review
10rule-builder 不设 Reviewer（v0.1），Architect 自行验收rule-builder
11固定 Agent 定义由 Architect 直接注入，不自行撰写 task spec两个 skill
12Assembly 分层合并：≤10 万字单次，>10 万字分组并行 + Reviewer 抽样contract-review
13修订合同 .docx 每处修订附批注（1–2 句修改原因）contract-review
14Reference 文件是参考示例，非强制模板contract-review
15修订人姓名在 bootstrap 确认，默认"审核方"contract-review
16复杂模式初设后默认暂停等待用户确认（可选自动继续）contract-review
17rule-builder 设有初步设计阶段（T-S01→T-S02+T-S03），与 contract-review 同位rule-builder
18T-S02/T-S03 交付标准因 skill 而异：contract-review 侧重可核验性与冲突检测，rule-builder 侧重商业逻辑理解与保护网分析两个 skill
