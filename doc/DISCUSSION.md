# 讨论与待解决问题

本文档记录项目当前正在讨论的设计问题、测试发现和待决策事项。已解决的问题移入 DESIGN.md 或 TECHNICAL-DESIGN.md，不再保留于此。

---

1、项目在确认用户立场的时候，要增加一个需确认的事项：在使用修订模式的时候使用什么名字来修订。且在后续实际通过修订模式改合同的时候，就要以该名字进行修订。

**分析**：完全合理。Word 修订模式中每处修订都标注作者名，如果由 python-docx 默认生成会是空白或系统用户名，交付给客户显得不专业。用户需要指定修订人姓名（如自己的名字、律所名称或具体律师名）。

**影响范围**：
- SKILL.md Bootstrap：在审核立场确认步骤中增加一问"修订人姓名"
- `_internal/architect-materials/shared-context.md`：新增 `reviser_name` 字段
- `schemas/shared-context.schema.md`：补充该字段定义
- 简单/复杂模式的 Revision Agent task spec：将修订人姓名作为要求写入，注明 docx-cli 设置 author 时使用该名字
- `references/task-revision.md`：示例中体现修订人姓名的使用

**实现要点**：python-docx 设置修订作者需要操作 XML（`w:ins` 元素的 `w:author` 属性），这比设置通用属性复杂。需要在 task spec 中明确告知 Revision Agent 这一技术要求。

**默认值**：用户如不指定，默认修订人姓名为"审核方"。

2、对于复杂模式，在采集用户立场/修订人信息时，要同步采集一个信息，是否在完成了初步设计后自动开始审查工作。若否，则在完成了初步设计后，在开始下一步审查工作前，要先等待用户确认后，再开始后续的工作。

**分析**：完全合理。复杂模式涉及大量 Agent 调用和 token 消耗，初步设计（T-S01/T-S02/T-S03/T-PR）让用户看到合同结构、关键商务条件、交叉引用关系和初步报告后，用户可能：

- 基于初步发现调整审查重点（如"重点关注第 5 章付款条款"）
- 发现合同与自己理解有差异，需要先内部讨论
- 决定没有必要继续深入审查（如合同风险可控）

强制自动继续会让用户失去这个决策窗口，且产生不必要的 token 消耗。默认"暂停等确认"是更安全的设计。

**影响范围**：
- SKILL.md Bootstrap：在立场确认环节增加此问题（"初步设计完成后是否自动继续，还是等您确认后再开始详细审查？"）
- `workflows/complex.md` 阶段 1.4（合同类型判断与规则匹配）之后、阶段 2 之前：插入条件分支——若用户选择"等待确认"，Architect 暂停流程，向用户呈现初设摘要（T-PR 报告的核心结论），等待用户指示后继续；若"自动继续"，直接进入阶段 2
- `doc/workflow.html`：复杂模式流程图中体现此决策节点

**默认值建议**：默认"等待确认"（opt-in to auto-continue）。原因：复杂合同的用户通常是高级律师，他们需要掌控审查方向；token 成本由用户承担，应由用户决定是否进入最昂贵的阶段。如果用户觉得每次都要确认太麻烦，可以选择"自动继续"。

**待讨论：暂停时向用户呈现什么？**

当用户选择"等待确认"，Architect 在初设完成后暂停流程，向用户发送一条消息等待指示。这条消息应该包含多少信息？有三个选项：

**选项 A — 极简通知**：仅告知"初设已完成"，请用户指示是否继续。

示例：
> 初步设计阶段已完成。已产出结构化合同（{N} 条）、主要合同条件报告、交叉引用报告和初步情况报告。是否继续进入详细审查阶段？

优点：简短，不消耗额外 token。
缺点：用户需要自己去读 T-PR 报告才能判断，相当于没有提供决策支持——用户原本就可以在 T-PR 产出后自己查看。

**选项 B — Architect 提炼摘要**：Architect 读三份初设文档和 T-PR 报告，提炼一份简短摘要，包含本次审查可能的核心风险和审查方向预估。

示例：
> 初步设计阶段完成。核心发现：
> - 合同类型：建设工程施工合同，约 {N} 条
> - 关键商务条件：合同价款 {金额}，工期 {天数}，违约金比例 {比例}
> - 识别到 {N} 处交叉引用，其中 {N} 处存在潜在冲突（条款 X 与条款 Y 对违约责任约定不一致）
> - 初步判断高风险区域：付款条件、变更管理、违约责任
>
> 建议的详细审查维度：付款条款、工期与延误、变更与签证、违约责任、争议解决。
>
> 是否继续进入详细审查阶段？您可以调整审查重点或指定重点关注领域。

优点：用户获得浓缩的决策信息，不需要自己去读 T-PR 报告。信息量适中，足够做"继续/调整/停止"的决策。
缺点：Architect 需要额外做一轮信息蒸馏（但这本就是 Architect 的核心能力，且已有 shared-context 和 reviewer-briefing 的编制经验）。

**选项 C — 直接呈现 T-PR 报告**：不单独提炼，告知用户"初设已完成，详见 preliminary-report.md"，等待指示。

优点：无额外工作。
缺点：T-PR 报告是面向用户的正式报告（2000-4000 字），包含了全部内容。用户暂停在这里想做的决策是"要不要继续、要不要调整方向"，而不是读一份完整的初步报告。信息过量反而降低决策效率。

**结论**：选项 A。用户是专业律师，清楚该看什么文件，不需要 Architect 额外提炼摘要。暂停时仅需告知初设完成、指明产出文件路径、请用户指示是否继续或调整方向。
