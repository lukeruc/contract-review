# 合同审核系统设计思路

## 项目目标

用户提供待审合同，系统交付两样成品：审核意见书 + 以修订模式标注修改的合同 .docx。

## 总体设计：EPC 三层模式

设计灵感来自工程项目的 EPC（Engineering, Procurement, Construction）总承包模式。核心思想：**总承包负总责，专业分包干专业活，复杂场景监理逐项核查质量，用户只需对接一个入口**。

> 技术实现命名：承包商 (Architect)、分包商 (Task Agent)、监理 (Reviewer)。

### 第一层：承包商层（Contractor）

承包商是用户唯一的交互入口。承包商不做第一手法律分析——不从零分析条款、不从零检索法规、不撰写法律意见或起草条款修改。承包商做的是管理加判断：读分包商产出、做决策、提炼信息、验收交付物。

职责：

1. **审核立场确认** —— 在进行任何工作之前，向用户确认审核立场：代表合同中的哪一方、审核目标、风险偏好。不同立场下同一条款的风险判断截然相反，这是不可逆的根本性决定。
2. **需求澄清（动态过程）** —— 贯穿整个项目周期。承包商在推动分包商执行的过程中，根据中间产出不断收敛和补充需求；当分包商反馈触及关键决策点时，向用户回报并要求指示。
3. **工作设计** —— 将整体交付物拆解为若干子工作项，明确每项工作的交付标准、输入输出接口、以及子工作之间的依赖关系。第一手法律分析全部委托分包商执行。
4. **信息蒸馏与分发** —— 从监理验证过的分包商产出中提取、归总关键信息，编制共享上下文和监理背景信息文档。输入是已验证的中间产物，不是原始合同，这是信息管理而非法律分析。
5. **交付组装与验收** —— 收集各分包商的产出，组装为审核意见书和修订后的合同，对照用户需求做最终验收。验收时可能读合同原文，但目的是对照核查，而非从零分析。

### 第二层：分包商层（Subcontractor）

分包商是承包商按任务需要创建的 sub-agent 实例，任务完成后销毁。承包商决定创建什么类型的分包商，系统提供常用模板（如审核 Agent、修订 Agent、检索 Agent 等），承包商可引用模板或自行定义。

职责：接收工作规格和交付标准，按要求产出工作产品，经监理审查通过后提交承包商验收。

### 第三层：工具层（Tools）

工具是分包商赖以获取信息和处理文件的设施。分包商在执行任务时按需调用——例如读取一份 docx 合同、检索某个法律问题、将修订标注导出为修订模式的 .docx。工具不做法律判断，只完成机械操作。

合同审核需要的工具涵盖：文档读取（Docx/PDF/OCR）、法律数据库检索、企业工商信息查询、网页搜索、合同文本结构化提取、Docx 生成与修订模式导出。

Agent 间通讯使用纯 Markdown。

## 工作流程：区分思想

EPC 是重型机制，不是所有合同都需要。系统收到待审合同后，首先将其转为 Markdown——无论后续走哪条路径，格式转化都是必经的第一步。格式转化是一个独立的 bootstrap 工具，在 EPC 机制启动前执行，不属于任何分包商。

随后，承包商通过工具获取该 Markdown 文件的字符数。阈值：**5000 字符**。不超过阈值走简单路径，超过走复杂路径。判断标准纯机械，不涉及 AI 判断。

在分发到简单或复杂路径之前，承包商必须完成一个关键步骤：**向用户确认审核立场**。承包商尝试从用户请求和合同文本中识别当事方，然后暂停流程请求用户确认代表哪一方、审核目标和风险偏好。这一步不可跳过——不同立场下同一条款的风险判断截然相反，无法事后修正。

### 简单模式：最小路径

合同短，承包商上下文充裕——不需要将任务肢解给多个分包商，也不需要监理。监理存在的唯一理由是保护承包商的上下文不被多分包商交付物淹没，简易模式下不存在这个压力。承包商读合同全文只做验收对照，不做第一手分析。

流程：承包商创建单个审核 Agent 产出审核意见 → 承包商对照合同全文和交付标准自行验收 → 通过后创建单个修订 Agent 完成合同修订稿 → 承包商对照已通过的审核意见书逐条核验修订内容，最终验收。

### 复杂合同：EPC 全机制

合同文本超出上下文窗口，不能直接丢给单个 Agent 做审核，必须先摸清全貌再做拆解。

承包商不亲自读合同。它创建分包商完成初步设计，产出四份文档：

1. **结构化合同文本** —— 将 Markdown 合同按条款层级、编号体系整理为结构化文本，是所有后续工作的信息底座。

结构化是所有复杂合同的共同步骤，任务内容恒定：识别层级、保留原文、标注异常。因此 Structure Task Agent 使用**固化定义**——system prompt 和交付标准预设，Architect 直接注入，不自行撰写 task spec。Agent 侧通过以下机制保障产出质量：

- **两遍处理**：第一遍只读不写，通篇识别编号体系、层级边界、异常点，建立全局心智模型；第二遍逐段输出结构化文本。
- **输出附带索引表**：结构化文本之外，额外输出一份索引表——每个章节对应的条款范围、条数、编号异常。生成索引的过程迫使 Agent 完成一次全局自检。
- **具体错误模式清单**：system prompt 中显式列明已知陷阱（编号跳跃是体系变更还是原文遗漏、附件清单被误读为条款、编号藏在长段正文中等），指名比笼统说"注意准确性"更有效。

索引表同时承载两个功能：Agent 生成时做自检，Reviewer 审查时做核查清单——Reviewer 按索引表逐条搜索原文验证，每次检查是一次精确匹配，不依赖注意力覆盖全文。
2. **主要合同条件报告** —— 提取关键商务条件（价格、交付、付款、期限等）和核心法律机制（违约责任、终止、管辖等），形成可核验的数据清单。
3. **合同内联系报告** —— 分析条款之间的引用、制约、补充、冲突关系，输出关联关系图和高风险交叉点。
4. **共享上下文** —— 承包商基于以上三份文档提炼的跨分包商共同基础：合同类型、适用法律、当事方身份与角色、核心交易结构摘要、贯穿全合同的关键术语定义、审核立场假设。

以上四份文档产出后经监理审查（见监理章节），审查通过后方可进入详细审查。

承包商从四份初设文档中提取监理后续工作所需的关键信息，编制**监理背景信息文档**——可核验数据清单、条款关系速查、高风险区域标注、审核立场与边界。此文档与共享上下文一并注入监理的阶段二上下文，使监理具备单点实质性审查能力。

后续详细审查和交付阶段按 EPC 标准流程运转：承包商派工（每个分包商注入共享上下文）、分包商执行、监理核查、承包商组装验收。

## 工作产品（Work Product）

分包商的生产对象是**可版本化、可追溯的中间产物**。承包商的派工接口体现为："将工作产品 X 从版本 vN 演进到 vN+1，满足标准 Y"。分包商之间的协作关系由承包商在工作设计中规划，体现在各任务的交付标准和依赖排序中。所有中间版本可追溯，便于回溯和迭代。

## 对话续接（Agent Continuity）

一次合同审查完成后，对话上下文已大量消耗。用户如果想追问细节（"上次说的第三条风险具体是什么""这个修改的法律依据在哪"），继续在原对话中成本高，新建对话则 Agent 对合同一无所知。

解决方案：在会话目录下创建 `agent/` 目录，作为新对话的入口点。当用户在 `agent/` 路径下新开 Claude Code 实例时，`CLAUDE.md` 自动加载，Agent 即刻进入对话状态。

### agent/ 目录结构

```
{会话目录}/
└── agent/                    # 用户新开 Claude Code 的入口
    ├── CLAUDE.md             # 自动加载的上下文快照
    └── index.md              # 全部工作产品文件索引
```

### CLAUDE.md 的内容

需要让新 Agent 同时拥有三样能力：知道合同是什么、知道审查发现了什么、知道想深入时去哪找文件。

- **合同速览**：合同名称、类型、当事方、核心交易结构（2-3 句）、适用法律、审核立场。来自 shared-context.md，承包商已提炼过
- **审查结论摘要**：总体风险评级、高/中/低风险数量、最重要的发现（3-5 条）。来自审核意见书执行摘要
- **修订概况**：修改了几条、涉及哪些条款范围
- **文件索引指引**：告知 Agent 去哪找完整信息——审核意见书、审核工作报告在 `../output/`，原始合同在 `../original/`，工作产品在 `../_internal/`
- **常见追问导航**：预判用户可能问的方向（"某条款改了什么措辞""审查时检索了哪些法条""对方不同意修改怎么谈"），每个方向标注应读取哪个文件
- **Agent 自我认知**：告诉新 Agent"你不是重新审合同，你是基于已有产出回答用户追问。所有法律分析已完成，你的工作是定位、解释和展开"

### index.md 的内容

按目录分组列出所有工作产物文件，每行含文件名、简短描述、产生者。新 Agent 由此了解项目全貌，无需 `find` 或猜测。

### 审核工作报告

除 agent/ 目录外，承包商同步产出 `output/audit-work-report.md` + `.html`——一份面向用户的审核工作报告。内容包括合同概要、工作流程记录、审核发现摘要、主要修改点逐条列表和文件索引。报告核心是"主要修改点"——让用户无需打开审核意见书全文即可了解最重要的修改内容和风险分布。篇幅适中，语言清晰不含法律黑话。

### 谁来做

承包商亲自编制以上全部内容——它经历了全流程，掌握合同全貌和审查结论，且信息蒸馏是其核心能力。不需要创建分包商。交付完成后执行，不阻塞主流程。

### 文件性质

`agent/` 不在 `_internal/` 下——`_internal` 是内部工作区，`agent/` 是用户直接使用的入口。`agent/CLAUDE.md` 不需要 YAML frontmatter schema，它是给 Claude Code 而非给另一个 Agent 读取的上下文文件。

## 承包商的决策上报机制

### 核心原则

承包商默认**自行决策**，上报用户是例外而非常态。目标用户是专业律师，需要的是成品而非选择题。

### 上报门槛

只有**同时满足**两个条件才提请用户决策：

1. **不可逆的实体性选择** —— 影响客户核心法律权益，一旦选定无法在不产生重大返工的前提下改弦更张。
2. **无合理默认解** —— 基于法律惯例、行业标准、风险最小化等原则，无法推导出一个明显更优的方案。

### 决策分类矩阵

```
                    法律技术判断          客户利益判断

无实质替代方案    →   自主执行           →  告知用户（FYI）

有实质替代方案    →   自主选择+记录      →  提请用户决策
                    推荐方案及理由         附带法律分析
```

承包商将用户视为"合伙人"而非"客户"——绝大多数操作性决策自行完成，仅在策略级分歧上上报。

## 监理 Agent（Supervisor）

监理是复杂模式专用的质量保障设施，受承包商委托核查分包商产出是否符合交付标准。简易模式下不启用监理，由承包商自行验收。

监理存在的唯一理由是保护承包商的上下文。复杂模式下承包商管理多分包商协作、做战略决策，如果每次分包商交付都要亲自逐条审查，上下文会被淹没。监理替代承包商完成日常审查，承包商只做最终验收。

监理的工作分为两个阶段，上下文随阶段递进。

### 在项目中的位置

```
用户 ←→ 承包商 ←→ 分包商 ←→ 工具层
            ↕
          监理
```

- 承包商直接向分包商下达工作（含交付标准），监理同步获得该交付标准。
- 分包商调用工具层完成工作，产出后提交监理审查。
- 监理逐条对照交付标准审查：通过则放行给承包商验收，不通过则退回分包商修订。此迭代对承包商透明。
- 承包商只看到监理放行的终版产物，做最终验收。

### 阶段一：初步设计审查

承包商创建分包商产出初设四份文档，同时下达工作要求（格式规范、章节覆盖范围、数据核验项）。监理是一个全新的 sub-agent 实例，上下文 = system prompt + 工作要求 + 分包商交付物。监理仅做形式审核：对照工作要求逐项检查格式是否正确、章节是否覆盖、数据引用是否可追溯。监理不做法律实质判断。

审查通过后，承包商基于初设文档产出共享上下文和监理背景信息文档。阶段一的监理实例随任务完成而销毁。

### 阶段二：工作审查

阶段二的每一次监理调用都是一个全新的 sub-agent 实例，与阶段一或其他任务的监理实例互不共享上下文。上下文 = system prompt + 共享上下文 + 监理背景信息文档 + 当前任务的交付标准。

监理背景信息文档为监理提供实质审查所需的参照系：可核验数据清单（金额、日期、比例）、条款关系速查、高风险区域标注、审核立场。监理可以核验分包商产出中的事实引用是否正确、条款关系是否与联系报告吻合、结论是否与合同基本结构明显矛盾。这使监理从纯形式审查升级到**单点实质性审查**——能发现明确的引用错误、数据错误和结构矛盾。监理不能发现的遗留盲区是跨分包商产出的语义矛盾（各自自洽但放在一起冲突），这属于未决事项中实质性审查 Agent 的范围。

监理的 system prompt 和核心判断立场所有实例相同，变化的是每实例独立注入的上下文信息。

### 定位：承包商的上下文防火墙

监理的首要功能是**保护承包商的上下文不被膨胀**。承包商要维持全局视野、管理多分包商协作、做战略决策——如果每次分包商交付都要逐条审查，判断能力就会退化。监理替代承包商完成"逐条对照交付标准"这个消耗上下文但判断难度可控的工作。

### 判断立场

- **默认信任分包商产出**。仅在发现明确错误时标注，不确定即不标注。
- 机械可判定项直接判通过或不通过；策略性内容监理不评，由承包商亲审。
- **循环上限为 3 次**。监理退回分包商修订最多 3 轮。3 轮后仍不通过，监理向承包商报告任务失败，附带失败记录——含每轮审查意见和分包商回应摘要。承包商据此决定调整交付标准、更换分包商、或亲自介入。

## 分包商间的一致性保证

横向一致性（多个分包商的产出之间不矛盾）靠共享上下文加依赖排序双机制保障。共享上下文在初设阶段产出（详见工作流程章节），注入所有详细审查分包商的工作上下文，确保各自阅读理解合同局部时基于一致的前提假设。依赖排序确保有依赖关系的工作串行，上游稳定版本作为下游输入；无交叉依赖的分包商可并行。

## 未决事项

1. **实质性审查 Agent（跨产品一致性审查）** —— 监理在阶段二已具备单点实质性审查能力（核验数据引用、条款关系、结构矛盾），但不能发现两个分包商各自产出自洽但放在一起语义冲突的跨产品矛盾。可能需要一个独立的审查 Agent，在所有分包商产出完成、承包商组装阶段介入。留待后续设计。

2. **共享上下文更新后已有批准产物的有效性** —— 承包商更新共享上下文后，此前基于旧上下文批准的产出是否仍然有效。留待后续设计。

3. **文件存储与管理系统** —— 工作产品的版本化与持久化、中间产物的追溯与比较，均需要一套文件管理基础设施。留待后续设计。
